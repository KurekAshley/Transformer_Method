# Contributing to the Transformer Method Canon

Thank you for your interest in contributing to the Transformer Method Repository.

This repository serves as the official canon of the Transformer Method and is dedicated to preserving, organizing, and evolving the transformational body of work developed by Kurek Ashley.

The purpose of this document is to ensure that all contributions strengthen the integrity, clarity, and effectiveness of the system.

---

# Before You Contribute

Before submitting any contribution, understand that the goal of this repository is not simply to collect information.

The goal is to preserve a coherent transformational system.

Every doctrine, framework, model, distinction, and methodology must support the larger architecture of the Transformer Method.

Contributions should prioritize:

- Clarity
- Accuracy
- Simplicity
- Practical application
- System integration
- Transformational value

---

# Core Principle

All contributions should align with the foundational principle of the Transformer Method:

> Transformation occurs when identity changes.

Any contribution that contradicts or undermines this principle will not be accepted into the canon.

---

# Types of Contributions

The following contribution types are welcomed.

## Doctrine Development

Expansion of existing doctrines.

Examples:

- Additional examples
- Clarifications
- Supporting explanations
- Practical applications
- Enhanced diagrams

---

## Framework Development

Creation or refinement of transformational models that support existing doctrine.

Examples:

- Visual frameworks
- Assessment models
- Coaching structures
- Learning pathways

---

## Documentation Improvements

Improvements to repository organization and readability.

Examples:

- Formatting
- Navigation
- Internal links
- Diagrams
- Documentation structure

---

## Research and Validation

Evidence, case studies, observations, and supporting material that strengthen existing doctrines.

---

# Contribution Process

All contributions should follow the process below.

## Step 1 — Identify the Need

Determine:

- What problem is being solved?
- What clarity is being added?
- What doctrine is being supported?

Contributions should serve a clear purpose.

---

## Step 2 — Align With Existing Doctrine

Review related doctrines before proposing changes.

Questions:

- Does this align with existing principles?
- Does this create contradiction?
- Does this strengthen the overall system?

If contradictions exist, refinement is required before submission.

---

## Step 3 — Submit Proposal

Prepare a proposal containing:

### Summary

A concise description of the contribution.

### Purpose

Why the contribution is valuable.

### Doctrine Alignment

Which doctrines support or relate to the contribution.

### Expected Impact

How the contribution improves understanding or application.

---

## Step 4 — Review

All contributions are reviewed for:

- Accuracy
- Clarity
- Consistency
- Integration
- Transformational value

Review does not guarantee acceptance.

---

## Step 5 — Canonical Approval

Only approved contributions become part of the official canon.

Final approval authority resides with the repository owner.

---

# Doctrine Creation Standards

New doctrines should include the following sections whenever applicable.

```text
Status
Definition
Core Principle
Models
Examples
Daily Practice
Diagnostic Questions
Master Model
Key Distinction
Canonical Statements
Related Doctrines
```

This structure ensures consistency across the repository.

---

# Writing Standards

All content should follow these principles.

## Be Clear

Write so concepts can be easily understood and taught.

## Be Practical

Every doctrine should have real-world application.

## Be Consistent

Use established terminology whenever possible.

## Be Concise

Remove unnecessary complexity.

## Be Transformational

Content should facilitate meaningful personal growth and change.

---

# Terminology Standards

When introducing terminology:

- Define terms clearly.
- Avoid unnecessary jargon.
- Use consistent language.
- Preserve established Transformer Method vocabulary.

Existing terminology should not be altered without strong justification.

---

# Doctrine Lifecycle

All new doctrines progress through the following stages.

```text
Concept
↓
Development
↓
Integration
↓
Canonization
```

Only doctrines that successfully complete this process become canonical.

---

# Reasons Contributions May Be Rejected

A contribution may be rejected if it:

- Contradicts existing doctrine
- Creates unnecessary complexity
- Lacks practical application
- Duplicates existing content
- Weakens system coherence
- Conflicts with foundational principles

Rejection is not necessarily permanent.

Contributors are encouraged to revise and resubmit when appropriate.

---

# Intellectual Property

By contributing to this repository, contributors acknowledge that accepted contributions become part of the Transformer Method Canon and may be modified, integrated, expanded, or adapted within the repository.

All repository content remains subject to the intellectual property policies defined by the repository owner.

---

# Contribution Philosophy

The purpose of contribution is not merely to add information.

The purpose of contribution is to strengthen transformation.

Every contribution should help people:

- Expand identity
- Elevate frequency
- Improve behaviour
- Create better results
- Become more than they were before

If a contribution serves that purpose, it belongs in the conversation.

> The purpose of contribution is not accumulation of knowledge. The purpose of contribution is the advancement of transformation.
